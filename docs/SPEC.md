# SPEC：保险条款智能问答系统（Insurance RAG）

> v1.0 · 2026-07-03 · 状态：待评审
> 关联文档：ARCHITECTURE.md（技术方案）、TODO.md（执行计划）

## 1. 问题陈述

保险代理人回答客户条款类问题时，需要在几十页的条款 PDF 中人工查找依据，单次查询耗时 5-15 分钟；凭记忆作答则存在误导销售的合规风险。本项目为代理人提供秒级、带条款原文引用、无依据即拒答的条款问答工具。第一用户是作者本人（保险副业日常使用），同时本项目作为作者转型 AI 应用开发工程师的代表作，需完整展示 RAG 工程能力（可插拔策略对比 + 量化评测）。

## 2. 目标

1. 条款问答准确可信：最优配置下 RAGAS faithfulness ≥ 0.90，评测集拒答题拒答率 100%
2. 策略对比可量化：Playground 支持 8 套核心配置组合，评测表能证明 structural 切片 + hybrid 检索 + rerank 相对 baseline 的提升幅度（预期 context_recall 提升 ≥ 10%）
3. 真实可用：作者本人每周实际使用 ≥ 3 次，单问 P95 延迟 < 5s
4. 可公开展示：线上 demo 链接 + GitHub 仓库 + 含评测数字的 README，两周内上线并进入简历

## 3. 非目标（v1 明确不做）

- **保费测算与计划书对比**：数据（费率表/计划书）需另行上传且属结构化查询范畴，v2 用 function calling 查表实现，v1 对保费类问题一律拒答并说明原因
- **C 端购买建议**：涉及销售合规红线，产品定位永久排除
- **多用户/权限体系**：v1 单用户使用，无登录
- **前端框架化**：v1 用静态 HTML + 原生 JS，Next.js 留给 v2
- **微调模型**：全部使用 API 模型，不训练

## 4. 用户故事

作为保险代理人：

- 我想用口语提问（"甲状腺结节能买吗"）并得到条款依据的回答，以便快速、合规地回复客户
- 我想在每条回答里看到条款原文出处（产品-条号-页码），以便复制给客户作为依据
- 当知识库里没有依据时，我想得到明确的"无法回答"而不是编造的答案，以便避免误导客户
- 我想上传新的条款 PDF 自动入库，以便扩充覆盖的产品

作为开发者（面试展示场景）：

- 我想勾选不同的切片/检索/重排/LLM 策略并左右对照同一问题的结果，以便直观展示每个组件的作用
- 我想一键跑完评测集并看到各配置的 RAGAS 分数表，以便用数字证明技术选型
- 我想看到每次回答的耗时与 token 消耗，以便讨论成本与延迟的权衡

## 5. 需求

### P0（缺一不可）

| # | 需求 | 验收标准 |
|---|---|---|
| R1 | PDF 解析入库（文本路径） | 给定 3 份 AIA 条款 PDF，`python scripts/ingest.py` 一条命令完成解析→双切片→元数据→索引；重复入库不产生重复数据 |
| R2 | 双切片策略 | fixed 与 structural 各生成独立 collection；structural 切片的 chunk 带「产品>章>条」层级元数据；随机抽 10 个 chunk 人工检查无条款被拦腰截断 |
| R3 | 混合检索 + 重排 | /ask 支持 vector/hybrid 切换与 rerank 开关；hybrid = 向量 + BM25 + RRF |
| R4 | 引用与拒答 | 回答中每个事实论断带 [产品-条号-页码] 标注，前端可点开对应原文 chunk；检索分数低于阈值时输出拒答话术；保费类问题被路由拦截并说明 |
| R5 | Playground 对照 | 前端可勾选 config，同一问题双配置左右对照，展示各自检索到的 chunk 与回答 |
| R6 | 评测 | 50 条评测集（五类问题×三档难度）；run_eval.py 跑完 8 套配置输出 results.json；RAGAS 四指标 + 拒答准确率渲染成表 |
| R7 | 流式与部署 | /ask 支持 SSE 流式输出；Dockerfile 单容器；HF Spaces 线上可访问 |

### P1（显著增强，时间允许即做）

| # | 需求 | 说明 |
|---|---|---|
| R8 | HyQE 增强索引 | 每 chunk 生成 2-3 个假设问题入索引，作为第三个 collection 参与对比 |
| R9 | VLM 表格解析 | 表格页/低质量页调用 Qwen-VL 转 Markdown；Playground 增加解析维度对比 |
| R10 | LLM 对比维度 | 最优检索配置上对比 deepseek-chat vs qwen-plus 的 RAGAS 分数与成本 |
| R11 | 多轮改写 | 指代消解（"那这款呢？"），基于会话历史补全问题 |

### P2（v2，写进 Roadmap 不实现）

计划书上传与多计划书结构化对比；费率表 function calling 查表；Next.js 前端 + Vercel；Milvus 迁移；多用户权限。

## 6. 成功指标

| 指标 | 目标 | 测量方式 |
|---|---|---|
| faithfulness（最优配置） | ≥ 0.90 | run_eval.py, 50 题 |
| context_recall 提升（最优 vs baseline） | ≥ +10% | 评测表对比 |
| 拒答题拒答率 | 100%（10/10） | 评测集拒答子集 |
| P95 端到端延迟 | < 5s | pipeline timings 日志 |
| 上线时间 | 开工后 ≤ 10 天 | demo 链接可访问 |
| 简历转化（滞后） | 项目进入简历并获得面试深挖 | 面试记录 |
