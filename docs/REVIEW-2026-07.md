## 评审框架

产品唯一的价值主张是「可信」（带引用、无据拒答、语料治理）。生产级 = 这些机制**在对抗与故障下依然成立，且被度量过**。按此标准，当前三根支柱的实际强度：拒答 = LLM 的自觉；治理 = 文件名的自觉；质量 = 一次都没跑过的评测。

## P0（阻断生产）

### A. 产品红线实际不成立

1. **默认配置无拒答兜底，安全开关客户端可控**。拒答阈值只在 rerank 成功分支生效（pipeline.py `_prepare`），而 `RagConfig` 默认 `rerank=False`，且 `/ask` 的 config 由调用方任意传。默认路径 = 向量 top-5 无分数下限直送 LLM；rerank 降级时同样无兜底。根因：Playground（实验）与生产（服务）两个身份共用一个 API。修法：生产入口锁定安全 config + vector/RRF 各设保守兜底阈值；自由 config 移到带鉴权的 Playground 端点。
2. **引用红线强制力为零**。用户问题原文拼进 prompt（一句"不要带编号"即可清空引用）；citations 为空只 warning 不拦截（pipeline.py）；LLM 编造的无效编号被静默替换为空串（citations.py）——幻觉论断保留、幻觉证据被抹平。修法：非拒答且引用为空 → 服务端改发拒答话术；无效编号计数进指标。
3. **数据治理 = 信任文件名**。deny 子串匹配可被 `Training_Deck.pdf`（下划线）绕过；白名单按文件名推导的 product 匹配，重命名即穿透；防御只在 `ingest_files` 单入口，`Indexer.index()` 无校验；`data/raw/` 内部材料与白名单产品混放。修法：白名单升级为「产品名 + 内容 sha256」双因子；Indexer 入口二次断言；内部材料移出目录。
4. **条款更新旧 chunk 永久残留**。全 ingest 无 delete 语义，chunk_id 为 `产品:策略:序号`，新版块数少于旧版时尾部旧块永留索引——会引用已废止条款，且不可观测。修法：按 product 清场式重入库 + BM25 补 delete。

### B. 评测「设计完成、从未运转」+ 方法论硬伤

- `eval/dataset.jsonl` 不存在、`eval/results/` 为空、167 条候选 ground_truth 全空——所有质量声明（faithfulness ≥ 0.90、拒答 100%）无实证。
- **幸存者偏差**：正常题被误拒后被踢出 RAGAS 打分池（harness.py `not r["refused"]` 过滤），而只有 rerank=on 会拒答 → 8 配置对比结构性偏向 rerank=on；误拒率无任何指标（拒答一切可拿满分）。
- **业务核心指标空白**：引用只存数量、chunk_id 不落盘、`source_chunk_id` 金标在手却不算 hit@k / citation_hit_rate——「引用是否指向正确条款位置」这一合规命门无度量，事后补算的原料也没留。

### C. 公开部署可用性

- **零鉴权 + 零限流 + LLM 无 max_tokens**：公网任意脚本可无限烧 embedding/rerank/LLM 账单。
- **SSE 无错误协议**：流中途异常 = 静默断流（HTTP 仍 200），前端无错误态、监控全盲；非流式路径裸 500 纯文本，前端不查 `resp.ok`。
- **假健康检查**：`/health` 无条件 ok；索引缺失时 collection 为空静默返回 [] → 全部问题走"礼貌拒答"，监控看一切正常。

### D. 入库静默失败

0 页解析成功照写 manifest（下次永远 skip）；单文件异常中止整批、无 failed 报告、CLI 恒 exit 0。

## P1（上生产前应修）

- 全同步架构共享 ~40 线程池，`/health` 同池——并发慢请求可拖死探针引发重启风暴；rerank 最坏 127s 才降级（30s×4 次重试）；rerank 响应越界 index 直接 500 绕过降级。
- BM25 内存快照与磁盘脱节（服务运行中 ingest 后 hybrid 融合两个版本语料）；pickle/manifest 非原子写，写坏则服务起不来；同产品多文件 chunk_id 互踩。
- 条款编号正则召回不足（「第X部/節」、无空格编号、全角、括号序号全漏）→ 引用降级轨，且主轨覆盖率无指标；<30 字短条款被吞并且继承错误条号；raw_text 链路断裂——展示的"条款原文"实为繁→简有损改写（opencc 应换 hk2s，UI 需标注）。
- 评测：judge 与被评模型同体（自评偏差），`--llm qwen-plus` 会连 judge 一起改坏；无逐题容错、8 配置跑完才落盘一次；同日同 commit 结果静默覆盖、不记 dirty/语料指纹/逐题分数；评测集按 structural chunk 生成且要求"可从片段回答"（对 fixed 不公平），fact 占 79%、comparison 结构性缺失。
- 测试盲区：生产必经的 `complete_with_usage` 分支零覆盖（所有 Fake 只实现 complete）；"LLM 抛异常时 /ask 行为"无测试；422 路径会真实构造 Chroma 客户端指向 repo data/index（conftest 未隔离路径）；中文端到端链路无 fixture（PDF 夹具仅 latin）。
- 供应链与前端：传递依赖无 lock；CI 缺 pip-audit/coverage/gitleaks/docker build；`esc()` 不转义引号却用于 `onclick` JS 字符串上下文（v2 用户上传上线即存储型 XSS）。
- 合规：无审计日志（争议时无法举证答了什么、引了哪条）；query 含客户 PII 直发第三方 API 无控制。

## P2（改进项，摘要）

拒答检测是"前 40 字含'无法回答'"的脆弱启发式（应在渲染前检测/结构化标记）；premium 路由误杀"退保能拿多少钱"类可答题、英文漏检；术语归一改写后的问题进了生成 prompt（应只用于检索）；FixedChunker 重叠常退化为零；BM25 每文件全量重建 O(N²)；低分拒答仍外发低分原文；SSE 缺反缓冲头；`stream=true` 与 response_model 契约失真（应拆端点）；api key 应换 SecretStr；RRF k=60 不可配；`/docs` 公开；镜像未 pin digest；无 /metrics（数据已算好只差暴露）；流式路径 usage 未采集；RAGAS 无中文适配；阈值 0.3 未校准且不支持 sweep；无 live 冒烟测试。

## 企业落地战略缺口（代码之外）

1. **身份与多租户**：鉴权、RBAC（按销售资质限产品线）、审计（HK IA 可回溯要求）是售前第一页。
2. **条款版本治理**：元数据无版本/生效日期维度；客户按投保时点适用对应版本条款——数据模型层的事，越晚改越贵。
3. **检索产品域隔离**：无 product filter，产品多了必然张冠李戴；需 query 侧消歧 + 检索侧过滤。
4. **供应商与数据主权**：HK 客户数据流向内地 API 是企业采购一票否决项；需私有化推理选项的验证路径。
5. **运维体系**：SLO/告警/成本看板/索引蓝绿发布/备份恢复。
6. **人在环**：/feedback 反馈回路 + bad case 采样落盘，是持续改进的原料。

## Playground 优化方案（2026-07-03 讨论定稿）

框架：Playground 价值 = 可控（变量）+ 可见（中间产物）+ 可比（量化 diff）。现状"可见"基本为零，是最大洼地。

1. **检索透明化**（优先级 1）：RetrievedChunk 携带各路 provenance（vector/BM25 的 rank+score、RRF 位次、rerank 分）——数据已在 `_hybrid` 内存中，只是被丢弃；前端逐 chunk 展示 + rerank 位次变化箭头；chunk 追踪（"预期命中的块在哪一站被丢"）。
2. **只检索模式**（优先级 1）：`/retrieve` 端点不调 LLM，调优迭代从 ~10s 降到亚秒且零生成成本。
3. **chunk 浏览器 + ingest 报告**（优先级 2）：入库时顺手产出静态报告（页级字符数/双栏判定/乱码率/繁简对照）；切片边界叠加视图（两策略并排，截断肉眼可见）；条号主轨覆盖率指标。
4. **参数可调 + 瀑布图**（优先级 3）：top_k/recall_k/refuse_threshold/RRF k 进 Playground config（生产端点锁定）；timings 渲染为瀑布 + token 成本。
5. **案例资产化**（优先级 4）：对比结果一键存档 → 导入 gen_eval_candidates 成为评测集候选；评测表下钻回放。
6. **解析多方案**（优先级 5）：预览先行（同页多解析器 diff + 质量指标，不入索引），胜者才建 collection；前置改造 collection 命名 → 显式 manifest 映射（同时解决评测语料指纹）。

安全边界：chunk 浏览器/检索 trace 等调试面只挂 Playground 面，随"生产与 Playground 分离 + 鉴权"一起落地，否则等于公网语料下载器。

## 建议行动顺序

1. 红线加固（G1，约 2-3 天）：空引用即拒发、兜底阈值+锁 config、白名单指纹、清场重入库、ingest 失败通道、原子写。
2. 评测闭环（G2，约 3-4 天含人工标注）：修偏差、补指标、judge 解耦、定稿 dataset、第一份结果入 git。**此前不对外声称任何评测数字。**
3. 公开部署前（G3，约 2-3 天）：鉴权限流熔断、SSE error + 异常 handler、/ready、lock + CI 加固。
4. 上线后一个月（G5 前半）：async 化、审计日志、/metrics、live 冒烟、raw_text 修复。
5. 企业化阶段（G5 后半）：SSO/RBAC、条款版本模型、私有化推理、Milvus/pgvector、蓝绿索引。

Playground 增强（G4）与上述并行推进：先做检索透明化与只检索模式（不碰索引、不花钱、当天见效）。
